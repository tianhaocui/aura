# Aura v0.6.0 PRD — AI-First 生产就绪

## 背景

v0.5.x 完成了 API 补全和开发者体验基础。v0.6.0 的目标是**让 AI 生成的代码一次跑通**，解决"从 demo 到生产"的最后一公里。

核心设计原则：优化 AI 一次正确率，不是优化人类少写几个字。

## 1. 错误信息自解释（P0）

### 动机

AI debug 的唯一输入是报错信息。如果报错说清楚了，AI 一轮就能修。

### 1.1 prop() 返回 null 时的诊断（dev mode）

```
// 之前
NullPointerException

// 之后
[Aura] prop("nacos.server-addr") returned null.
  Checked: env NACOS_SERVER_ADDR (not set), sysprop nacos.server-addr (not set),
           aura.properties (not found), aura-dev.properties (not found).
  Hint: use app.set("nacos.server-addr", value) or set env NACOS_SERVER_ADDR.
```

实现：prop() 返回 null 时，dev mode 下打 WARN 日志列出完整查找路径。

### 1.2 路由匹配失败的诊断（dev mode）

```
// 之前
404 Not Found

// 之后 (dev mode JSON body)
{
  "error": "No route matched: POST /api/user",
  "hint": "Did you mean: POST /api/users?",
  "registered": ["GET /api/users", "POST /api/users", "GET /api/users/{id}"]
}
```

实现：404 时在 dev mode 下返回最接近的路由（Levenshtein 或前缀匹配）。

### 1.3 services() 依赖解析失败

```
[Aura] Cannot create Engine:
  Constructor requires 7 params: [ScoreService, MatchService, OddsService, ...]
  Missing: ScoreService — no bean of this type registered.
  Registered beans: [Db, MatchService, EngineConfig]
  Hint: add ScoreService.class to services() or register an instance with app.register().
```

### 1.4 类型不匹配

```
[Aura] Cannot inject param #3 of Engine constructor:
  Expected: OddsService
  Found: OddsServiceImpl (not assignable — did you forget to implement the interface?)
```

## 2. API 语义零歧义（P0）

### 2.1 ctx.raw(body) — 已在 v0.5.5 实现

### 2.2 props(prefix) 走 resolve()

当前 `props("nacos.")` 只扫 props map，不查 env/sysprop。修复后和 `prop(key)` 行为一致：

```java
public Map<String, String> props(String prefix) {
    Map<String, String> result = new LinkedHashMap<>();
    for (Map.Entry<String, String> entry : props.entrySet()) {
        if (entry.getKey().startsWith(prefix)) {
            result.put(entry.getKey(), prop(entry.getKey())); // 走 resolve
        }
    }
    return result;
}
```

`props(prefix, Record.class)` 同理——ConfigBinder 接收的 map 应该是 resolve 后的值。

### 2.3 BeforeBuilder 链式完整性

加 `.and()` 方法回到 Aura，不需要加 32 个代理方法：

```java
app.before(authHandler).exclude("/health", "/login").and()
   .exception(ValidationException.class, handler)
   .get("/api/users", handler);
```

## 3. services() 自动装配（P1）

### 动机

AI 加一个 Service，当前要改 3 处（new、传参、注册）。`services()` 降到 1 处。

### API

```java
Aura.create()
    .register(db)                                          // 特殊组件：手动
    .register(app.props("engine.", EngineConfig.class))    // 配置：手动
    .services(
        ScoreService.class,
        MatchService.class,
        OddsService.class,
        AdapterService.class,
        Engine.class              // ← AI 加组件：只加这一行
    )
    .start();
```

### 行为

1. `services(Class...)` 收集类列表，不立即实例化
2. `start()` 时：
   - 分析所有 Service 构造器参数类型
   - 构建依赖 DAG
   - 拓扑排序
   - 按序实例化（构造器参数从已注册 bean 中按类型匹配）
   - 实例化后自动 `register()`
3. 循环依赖 → 启动报错（含依赖链）
4. 参数匹配不到 → 启动报错（含已注册列表 + hint）

### 匹配规则

- 精确类型匹配优先
- 接口/父类匹配次之
- 同类型多实例 → 报错，提示用 `register("name", instance)` + `@Named`
- String/int/boolean 等基础类型 → 不匹配，必须通过 Record config 注入

### 构造器选择

- 唯一 public 构造器 → 使用它
- 多个 public 构造器 → 报错（"ambiguous constructor, keep only one"）
- 无 public 构造器 → 报错

### 示例：AI 加一个新 Service

```java
// Step 1: AI 写 Service 类
public class NotificationService {
    private final Db db;
    private final Engine engine;

    public NotificationService(Db db, Engine engine) {
        this.db = db;
        this.engine = engine;
    }
}

// Step 2: AI 在 services() 加一行
.services(
    ...
    Engine.class,
    NotificationService.class   // ← 就这一行
)
```

AI 不需要知道 db 和 engine 的变量名、创建顺序、在哪一行。

### 生命周期

- 实现 `Closeable` / `AutoCloseable` 的 Service → stop 时自动 close（逆序）
- 实现 `Reloadable` 的 Service → 配置变更时自动调 reload()

## 4. aura-nacos 插件（P2）

### 动机

380 行 Nacos 样板代码，AI 每次生成都可能不同。内置标准实现。

### API

```java
Aura.create()
    .plugin(new NacosPlugin(app.prop("nacos.server-addr"), "my-service"))
    .start();
```

### 功能

- 服务注册（onStart 后自动注册，stop 时注销）
- 心跳保活
- 配置拉取（指定 dataId/group → 写入 app props）
- 配置变更监听（变更时触发 Reloadable 组件 reload）

### 模块结构（独立仓库）

插件作为独立仓库维护，不放在 aura 主仓库：

```
tianhaocui/aura          — 核心框架 (core/web/db/dev)
tianhaocui/aura-nacos    — Nacos 插件 (注册/配置/心跳)
```

理由：
- 独立版本号（跟 nacos-client 版本走，不绑定 aura 发版节奏）
- 用户按需引入（不污染主仓库依赖树）
- 第三方可参照写自己的插件（aura-redis、aura-kafka）
- 主仓库保持纯净

aura-dev 留在主仓库（和核心强耦合）。

```
aura-nacos/
├── src/main/java/io/aura/nacos/
│   ├── NacosPlugin.java          // AuraPlugin 实现
│   ├── NacosRegistry.java        // 服务注册/注销/心跳
│   └── NacosConfigLoader.java    // 配置拉取/监听/触发 Reloadable
└── pom.xml                       // 依赖 nacos-client + aura-core
```

## 5. Reloadable 配置热更新（P2）

### 动机

配置变更时手动维护回调链容易漏。提供显式接口替代。

### API

```java
public interface Reloadable {
    void reload(Aura app);
}

// 组件实现
class ApiKeyFilter implements Reloadable {
    private volatile String apiKey;

    @Override
    public void reload(Aura app) {
        this.apiKey = app.prop("api.key");
    }
}
```

### 行为

- `services()` 注册的组件如果实现 Reloadable，框架自动收集
- 配置变更时（Nacos 推送、手动触发 `app.reloadConfig()`），框架按注册顺序调用所有 Reloadable
- 不是 @RefreshScope 代理魔法——对象不会被替换，只是调一个方法

## 配置优先级（明确文档化）

最终优先级（从高到低）：

```
1. 启动参数 --key=value
2. 环境变量 (KEY_NAME)    ← replace('.','_').replace('-','_').toUpperCase()
3. System Property (-Dkey=value)
4. app.set(key, value)
5. aura-{env}.properties
6. aura.properties
7. 代码默认值
```

`prop(key)`、`props(prefix)`、`props(prefix, Class)` 全部遵守此优先级。

## 补充说明

### @Path 类与 services 体系

`scan()` 扫到的 `@Path` 类自动走构造器注入（和 `services()` 相同逻辑）。不需要同时写两处：

```java
app.scan("com.myapp")       // @Path 类自动注入构造器参数 + 注册路由
app.services(Engine.class)  // 非路由的业务组件
```

### start() 前后状态

```java
Aura app = Aura.create();
app.prop("key");              // ✅ 可用（props 文件已加载）
app.getBean(Db.class);       // ❌ null（还没注册）

app.register(db);
app.getBean(Db.class);       // ✅ 立即可用

app.services(Engine.class);
app.getBean(Engine.class);   // ❌ null（start 时才实例化）

app.start();
app.getBean(Engine.class);   // ✅ 可用
```

**关键规则：`register()` 和 `services()` 的调用顺序不影响依赖解析。所有 resolve 发生在 `start()` 时。**

AI 不需要操心"register 要写在 services 前面"。只要 start() 前注册了，就能被 services 里的构造器找到。以下两种写法等价：

```java
// 写法 A
app.services(Engine.class)
   .register(db)
   .start();

// 写法 B
app.register(db)
   .services(Engine.class)
   .start();
```

### services 启动失败行为

任何一个 Service 实例化失败 → 打印完整错误信息（含依赖链） → `System.exit(1)`。

不允许"部分启动"。要么全部就绪，要么不启动。

### NacosPlugin 配置方式

无参构造器，自动从 props 读取：

```java
app.plugin(new NacosPlugin());

// aura.properties 里配
nacos.server-addr=127.0.0.1:8848
nacos.service-name=my-service
nacos.namespace=public
```

不让用户传参到构造器里。配置统一走 props 体系，env 可覆盖。

### 多 profile / env 来源

优先级：`--env=staging` 启动参数 > `AURA_ENV` 环境变量 > `aura.env` properties 配置 > 默认 `dev`

```
AURA_ENV=staging → 加载 aura.properties + aura-staging.properties（后者覆盖前者）
```

## API 唯一路径原则

**0.x 版本不做向后兼容。AI 只有一种写法。**

理由：
- 0.x = API 不稳定窗口期，用户应预期会变
- 两种等价写法并存 = AI 必然混用 = 更多 bug
- 唯一路径 = AI 零决策成本

### 语义分工

```java
register(instance)       // 基础设施：已创建的实例（Db、Config Record、第三方 client）
services(Class...)       // 业务组件：框架自动创建（构造器参数自动匹配）
```

- 参数类型不同（instance vs Class），AI 不会搞混
- 文档/示例/AI_GUIDE 只展示唯一推荐模式
- 不保留"两种等价写法"

### AI 生成的唯一模式

```java
Aura.create()
    .register(Db.create(app.prop("db.url"), ...))
    .register(app.props("engine.", EngineConfig.class))
    .services(
        ScoreService.class,
        MatchService.class,
        Engine.class
    )
    .start();
```

- `register()` = 基础设施，需要配置/外部创建的东西
- `services()` = 业务组件，纯依赖注入
- AI 加组件 = services 列表加一行，改 1 处

## 非目标

- 不做注解扫描 @Component/@Service（违背显式原则）
- 不做 @RefreshScope 代理替换（违背无代理原则）
- 不做 @Autowired 字段注入（构造器即依赖）
- 不做 @ConditionalOnXxx（显式 if 更清晰）
- 不做 Module 分组（对 AI 无价值）
- 不做两种等价 API 并存（AI 唯一路径原则）
