## 1. ctx.raw(body) — 原始响应写入

### 动机

`ctx.text()` 无条件覆盖 Content-Type 为 `text/plain`。当开发者需要自定义 Content-Type（如 `text/csv`、`application/xml`）时，`ctx.header("Content-Type", "text/csv")` 设好后调 `ctx.text(data)` 又被覆盖掉了。

目前没有"只写 body 不动 Content-Type"的方法。

### API

```java
// 先设好 Content-Type，再写 body
ctx.header("Content-Type", "text/csv; charset=utf-8");
ctx.raw(csvContent);

// 或者不设 Content-Type（Undertow 默认 application/octet-stream）
ctx.raw(someData);
```

### 实现

```java
// Context.java
@Override public void raw(String body) {
    exchange.getResponseSender().send(body);
}

// MockContext.java
@Override public void raw(String body) {
    responseBody = body;
}

// BaseContext.java (接口)
void raw(String body);
```

### 对比

| 方法 | Content-Type | 用途 |
|------|-------------|------|
| `ctx.json(obj)` | 强制 application/json | JSON 序列化 |
| `ctx.text(str)` | 强制 text/plain | 纯文本 |
| `ctx.html(str)` | 强制 text/html | HTML |
| `ctx.raw(str)` | **不设/不覆盖** | 自定义格式（CSV、XML 等） |

## 2. prop() 环境变量 key 转换 bug

### 问题

`app.prop("nacos.server-addr")` 转换环境变量 key 时只替换 `.` → `_`，不处理 `-`：

```
nacos.server-addr  →  NACOS_SERVER-ADDR  (实际查找的 key，错误)
用户设的环境变量   →  NACOS_SERVER_ADDR  (匹配不上)
```

导致环境变量看起来"不生效"，fallback 到 properties 文件值。

### 修复

```java
// Aura.java prop() 方法
public String prop(String key) {
    return resolve(key, key.replace('.', '_').replace('-', '_').toUpperCase());
}
```

一行修复。同时检查 `applyFrameworkProps()` 里其他 `resolve()` 调用是否有同样问题。

## 3. app.before() 路径排除

### 动机

全局 `app.before(authHandler)` 会拦截所有路由，包括 `/health`、`/api/adapter/health` 等不需要认证的端点。目前只能在 handler 里写 if 判断跳过：

```java
// 现在：丑
app.before(ctx -> {
    if (ctx.url().equals("/health")) return;  // 手动排除
    if (ctx.url().startsWith("/api/adapter/health")) return;
    checkAuth(ctx);
});
```

### API 设计

```java
// 方案 A：exclude 方法（推荐）
app.before(authHandler).exclude("/health", "/api/adapter/health");

// 方案 B：路径匹配式 before
app.before("/api/**", authHandler);  // 只匹配 /api/ 下的路由
```

方案 A 更贴合现有代码风格（before 是全局的，排除特例）。

### 实现思路

`app.before()` 返回一个 `BeforeBuilder`，支持 `.exclude(String... paths)`。内部包装为带路径判断的 handler：

```java
public class BeforeBuilder {
    private final BaseHandler handler;
    private final List<String> excludes = new ArrayList<>();

    public BeforeBuilder exclude(String... paths) {
        excludes.addAll(List.of(paths));
        return this;
    }

    BaseHandler build() {
        if (excludes.isEmpty()) return handler;
        return ctx -> {
            String path = ctx.url();
            for (String ex : excludes) {
                if (path.equals(ex) || (ex.endsWith("*") && path.startsWith(ex.substring(0, ex.length() - 1)))) {
                    return;
                }
            }
            handler.handle(ctx);
        };
    }
}
```

## 4. onStart 执行时机修复

### 问题

当前调用顺序：

```java
// Aura.start()
selfCheck();
fireStart();         // ← onStart hooks 在这里执行
starter.start(this); // ← Undertow server.start() 在这里
```

`onStart` 在端口监听**之前**触发。如果用户在 onStart 里注册 Nacos/Consul，服务发现系统会把流量导过来，但端口还没就绪 → connection refused。

### 修复

拆分为两个时机：

```java
// 方案 A（最小改动）：把 fireStart 移到 server 启动之后
selfCheck();
starter.start(this);  // 先启动服务器
fireStart();          // 端口就绪后再触发 hooks
```

```java
// 方案 B（更精细）：新增 onReady 回调
app.onStart(a -> { /* 启动前，初始化资源 */ });
app.onReady(a -> { /* 端口就绪后，注册服务发现 */ });
```

**建议方案 A** —— 简单，且 onStart 的语义"应用启动完成"本来就应该是端口就绪之后。如果有人需要在端口监听前初始化资源（如 DB），应该用 plugin 或直接写在 start() 前面。

### Breaking Change 风险

如果有用户在 onStart 里做的事依赖"服务器还没启动"的状态（几乎不存在），会受影响。实际上所有 onStart 使用场景都期望"应用已就绪"。低风险。

## 5. 配置优先级文档化 + 统一

### 当前实际优先级

```java
private String resolve(String propKey, String envKey) {
    String envVal = System.getenv(envKey);   // 1. 环境变量
    if (envVal != null) return envVal;
    return props.get(propKey);               // 2. props map (含文件 + set())
}
```

props map 的加载顺序（后加载覆盖先加载）：
1. `aura.properties`（默认配置）
2. `aura-{env}.properties`（环境配置覆盖）
3. `app.set(key, value)`（代码设置覆盖）
4. `--key=value` 启动参数（覆盖所有）

### 需要统一的问题

- 缺少 System Property (`-Dkey=value`) 支持
- 文档里没有明确声明优先级

### 最终优先级（从高到低）

```
1. 启动参数 --key=value
2. 环境变量 (KEY_NAME)
3. System Property (-Dkey=value)     ← 新增
4. app.set(key, value)
5. aura-{env}.properties
6. aura.properties
7. 代码默认值
```

### 实现

```java
private String resolve(String propKey, String envKey) {
    // 1. 环境变量（优先级最高的外部配置）
    String envVal = System.getenv(envKey);
    if (envVal != null) return envVal;
    // 2. System Property
    String sysProp = System.getProperty(propKey);
    if (sysProp != null) return sysProp;
    // 3. props map（含文件 + set + 启动参数，已按加载顺序覆盖）
    return props.get(propKey);
}
```

### 文档

在 AI_GUIDE.md 的 Configuration 章节明确写出优先级表。
