# Aura v0.5.3 PRD — AI 编码体验优化

## 背景

v0.5.2 修复了 daemon thread 和 multipart upload 问题。本版本聚焦于让 AI 编码助手使用 Aura 写代码时更高效、更不易出错。核心思路：减少 AI 需要做的判断和记忆，让"正确写法"只有一种。

## 3. 统一异常处理 app.exception()

### 动机

AI 写接口时不确定项目的错误响应格式，导致每个 handler 里都写 try-catch + 手动构造错误 JSON。全局 exception handler 让 AI 只需 throw，由框架统一处理。

### 现状

`router.exception(Class, handler)` 已存在，但需要嵌套在 `app.routes(r -> ...)` 中注册，AI 不容易发现。

### API 设计

在 `Aura` 类新增便捷方法，委托到内部 router：

```java
// 按异常类型注册（推荐）
app.exception(IllegalArgumentException.class, (e, ctx) -> {
    ctx.status(400).json(Result.fail(400, e.getMessage()));
});

// 兜底处理所有异常
app.exception(Exception.class, (e, ctx) -> {
    ctx.status(500).json(Result.fail(500, "Internal error"));
});
```

### 行为

- 委托到 router.exception()，行为与现有机制一致
- handler 或 middleware 抛出未捕获异常时触发
- 未注册 exception handler 时保持现有行为（IllegalArgumentException → 400，其他 → 500）
- 不捕获 Error 类型
- 框架默认已处理 IllegalArgumentException 和 ValidationException → 400，用户注册会覆盖默认行为

## 4. ctx.bodyOrThrow(Class)

### 动机

AI 写接口经常漏掉 null check 或校验步骤。合并为一个调用，解析失败或校验不通过直接抛异常（被全局 handler 兜住）。

### API 设计

```java
app.post("/users", ctx -> {
    CreateUserReq req = ctx.bodyOrThrow(CreateUserReq.class);
    // req 一定非 null 且已通过 @Validate 校验
});
```

### 行为

- body 为空或解析失败 → 抛 IllegalArgumentException("Request body is required")
- 自动调用 BeanValidator.validate()，校验失败抛异常（同时覆盖注解校验和 Validatable 接口）
- 异常消息包含具体校验错误信息
- MockContext 中同样可用（确保测试不踩坑）

## 5. ctx.queryRequired() / ctx.pathInt()

### 动机

AI 经常忘记做参数非空校验或类型转换。框架提供强类型方法，缺失时直接抛异常。

### API 设计

```java
app.get("/users/{id}", ctx -> {
    int id = ctx.pathInt("id");              // 解析失败抛异常
    String name = ctx.queryRequired("name"); // 缺失抛异常
});
```

### 方法列表

| 方法 | 行为 |
|------|------|
| queryRequired(name) | query param 缺失或空白时抛 IllegalArgumentException |
| pathInt(name) | path param 转 int，失败抛异常 |
| pathLong(name) | path param 转 long，失败抛异常 |

## 7. Profile 自动加载

### 动机

用户经常需要 dev/prod 不同配置，但 Aura 只加载 `aura.properties`。AI 不知道该怎么做环境差异化配置。

### 行为

- `Aura.create()` 时先加载 `aura.properties`
- 然后根据 `aura.env`（来自环境变量 `AURA_ENV` 或 aura.properties 中的值）自动加载 `aura-{env}.properties`
- env-specific 配置覆盖默认配置
- 文件不存在时静默跳过（不报错）
- 加载顺序：`aura.properties` → `aura-{env}.properties` → 环境变量 → 启动参数

### 示例

```
# aura.properties（默认）
aura.port=8080
db.url=jdbc:mysql://localhost/dev_db

# aura-prod.properties（生产覆盖）
aura.port=80
db.url=jdbc:mysql://prod-host/prod_db
```

## 8. 启动错误优化

### 动机

端口被占用时直接抛 `java.net.BindException` 堆栈，新手和 AI 都看不懂。应输出友好提示。

### 行为

- 捕获 `BindException`（Address already in use）
- 输出日志：`ERROR: Port {port} is already in use. Use --port=XXXX or set AURA_PORT to change.`
- 程序优雅退出（`System.exit(1)`），不打印完整堆栈

## 9. app.before() / app.after() 便捷方法

### 动机

跟 #3 同一问题：`before()` / `after()` 只能在 `app.routes(r -> { r.before(...) })` 里注册，AI 不容易发现。提供 app 级别的便捷入口。

### API 设计

```java
app.before(ctx -> log.info("{} {}", ctx.method(), ctx.url()))
   .before(Aura.requireAuth())
   .after(ctx -> log.info("done"))
   .routes(r -> { ... });
```

### 行为

- Aura 类内部维护 beforeHandlers / afterHandlers 列表
- start() 时将 app 级别的 handler 插入 router 的 beforeHandlers 头部
- 执行顺序：app.before → router.before → group.before → handler → group.after → router.after → app.after
- 支持链式调用

## 6. 文档与配套更新

### 目标

Aura 框架足够简单，AI 应该读一次文档就能正确生成代码，不需要试错。当前主要痛点不是功能缺失，而是 AI 无法发现已有功能。

### AI Guide 必须覆盖的主题

AI Guide 是 AI 编码助手的唯一参考源，必须做到"一文读完即可工作"。以下主题必须明确写出：

| 主题 | 需要说明的内容 |
|------|--------------|
| 路由注册 | app.get/post/put/delete 用法，路径参数 {id} 语法 |
| 请求参数 | ctx.query()、ctx.queryRequired()、ctx.pathInt() 完整对照表 |
| 请求体 | ctx.body(Class) vs ctx.bodyOrThrow(Class)，Record 类自动 JSON 反序列化规则 |
| 响应输出 | ctx.json() / ctx.text() / ctx.html() / ctx.status() 用法 |
| 中间件 | app.before() / app.after() 用法和执行顺序 |
| 异常处理 | app.exception() 注册方式，框架默认行为（400/500），throw 即可不需 try-catch |
| 参数校验 | BeanValidator + Validatable 接口，@Validate 注解，校验失败自动 400 |
| DI 容器 | app.register() / app.getBean() 手动注册模式，不是自动扫描 |
| 配置 | aura.properties 加载规则，app.prop() 读取，环境变量优先级 |

| 标准响应 | Result.ok() / Result.fail() 用法 |
| 测试 | TestClient 用法，MockContext 行为 |

### 约定必须显式声明（不能让 AI 猜）

- Record 类作为请求体：自动通过 fastjson2 反序列化，字段名 = JSON key
- Record 类作为响应体：自动序列化为 JSON
- query param 绑定：必须用 ctx.query("name") 手动获取，不做自动绑定
- path param 绑定：必须用 ctx.path("id") 手动获取
- DI 是手动注册，不是自动扫描（区别于 Spring）
- 配置文件只支持 .properties 格式，不支持 YAML
- 静态文件路径是 classpath 相对路径
