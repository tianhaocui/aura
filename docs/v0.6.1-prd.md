## 2. services() 里 @Path 类自动注册路由

### 动机

AI 加一个 Controller 需要改 2 处（services + routes），容易漏。@Path 注解本身就是显式声明路由前缀，放在 services() 里应该自动注册路由。

### 行为

```java
app.services(
    UserService.class,       // 无 @Path → 纯 DI
    UserController.class,    // 有 @Path("/api/users") → DI + 路由自动注册
    AuthController.class     // 有 @Path("/api/auth") → DI + 路由自动注册
)
```

规则：
- services() 里的类有 @Path → 构造器注入 + 自动注册路由
- services() 里的类无 @Path → 纯构造器注入
- 同一个类同时出现在 scan() 和 services() → 启动报错

### 为什么不隐式

@Path 注解写在类的第一行，是显式的路由声明。AI grep `@Path` 就能找到所有路由来源。这和 Spring 的 classpath 扫描不同——用户显式把类列在 services() 里了。

### 认证模式

不加 @Auth 注解。用路径规则集中管理：

```java
app.before(Aura.requireAuth())
   .exclude("/health", "/api/auth/login", "/api/auth/register");
```

### 实现

start() 里 services 实例化完后，检查 @Path 注解并注册路由：

```java
for (Object bean : newlyCreatedServices) {
    if (bean.getClass().isAnnotationPresent(Path.class)) {
        ServiceRegistrar.register(bean, router);
    }
}
```

### scan() 废弃

`scan()` 标记为 `@Deprecated`。services() 是唯一推荐方式。

## 2. controllers(Class...) — 路由组件注册

### 动机

`services()` 方法名不暗示路由行为。如果 @Path 类放在 services() 里自动注册路由，行为从代码表面看不出来。

### 方案

新增 `controllers(Class...)` 方法，语义明确：DI + 路由注册。

```java
Aura.create()
    .register(db)
    .register(app.props("engine.", EngineConfig.class))
    .services(
        UserService.class,         // 纯业务逻辑，DI only
        OrderService.class
    )
    .controllers(
        UserController.class,      // DI + @Path 路由注册
        OrderController.class,
        AuthController.class
    )
    .start();
```

### 语义分工（最终形态）

```
register(instance)        — 基础设施：已创建的实例（Db、Config Record）
services(Class...)        — 业务组件：纯构造器注入，不碰路由
controllers(Class...)     — 路由组件：构造器注入 + @Path/@Get/@Post 路由注册
```

三个方法，三个语义，互不重叠。

### AI 四问检验

| 问题 | 答案 |
|------|------|
| AI 能从方法名猜对行为？ | ✅ controllers = 处理 HTTP 请求的组件，必然有路由 |
| AI 加 Controller 改几处？ | 1 处（controllers 列表加一行） |
| 有等价写法吗？ | 无。services ≠ controllers，语义不重叠 |
| 出错时好排查？ | 启动日志打印 `Registered controller: UserController (3 routes)` |

### 实现

- `controllers(Class...)` 收集到 `controllerClasses` 列表
- `start()` 时走 ServiceResolver 做构造器注入（和 services 共用逻辑）
- 注入完后调 ServiceRegistrar.register(instance, router) 注册路由
- 如果 controllers 里的类没有 @Path 注解 → 启动报错：
  ```
  [Aura] UserService is in controllers() but has no @Path annotation.
  Move it to services() if it's not a route handler.
  ```
- 如果同一个 Class 同时出现在 services() 和 controllers() → 启动报错

### scan() 废弃

`scan()` 标记为 `@Deprecated`。`controllers()` 是它的显式替代。文档不再出现 scan()。

### 行为

```java

)
```

### 实现

```java
for (Object bean : newlyCreatedServices) {

    }
}
```

### 认证方案

不加 @Auth 注解。用路径规则集中管理：

```java
app.before(Aura.requireAuth())
   .exclude("/health", "/api/auth/login", "/api/auth/register");

app.services(
    AuthController.class,    // @Path("/api/auth") — login/register 被 exclude
    UserController.class,    // @Path("/api/users") — 需要认证

)
```

AI 加新 Controller：services() 加一行。不需要认证则 exclude 加一行。

## 3. `r.get(path, Class, "method")` 路由重载

### 动机

升级 0.6.0 后路由里每行都是 `app.getBean(XxxService.class).method(ctx)`，重复 50+ 次。

### API

```java
// 之前（啰嗦）
app.routes(r -> {
    r.get("/users", ctx -> ctx.json(app.getBean(UserService.class).list()));
    r.post("/users", ctx -> ctx.json(app.getBean(UserService.class).create(ctx)));
});

// 之后（简洁）
app.routes(r -> {
    r.get("/users", UserService.class, "list");
    r.post("/users", UserService.class, "create");
});
```

### 实现

Router 加重载，内部从 app registry 获取实例：

```java
public BaseRouteBuilder get(String path, Class<?> serviceClass, String method) {
    Object instance = app.getBean(serviceClass);
    if (instance == null) {
        throw new IllegalStateException("No bean of type " + serviceClass.getSimpleName() 
            + " registered. Add it to services() or register() before routes().");
    }
    return get(path, instance, method);
}
```

注意：这要求 routes() 在 services() resolve 之后执行。当前 start() 顺序是 services resolve → route compilation，符合。

但如果用户在 start() 之前调用 `app.routes(r -> r.get("/x", MyService.class, "m"))`，此时 bean 还不存在。需要延迟到 start() 时 resolve。

实现方式：routes lambda 本身就是延迟执行的（start() 时才 accept），所以 getBean 在 start() 时调用，services 已经 resolve 完了。没问题。

## 4. @Named 解决同类型多实例歧义

### 动机

findBean 检测到同类型多实例会报错，但用户没有消解手段。

### API

```java
// 注册时命名
app.register("portal", new ClientCredentialsTokenProvider(...));
app.register("login", new LoginTokenProvider(...));

// 构造器里用 @Named 区分
public class AdapterService {
    public AdapterService(@Named("portal") TokenProvider portal, Db db) { ... }
}
```

### 实现

ServiceResolver.findBean() 增加 @Named 检查：

```java
private static Object findBean(Class<?> type, Parameter param, Map<Class<?>, Object> registry, Map<String, Object> namedRegistry) {
    Named named = param.getAnnotation(Named.class);
    if (named != null) {
        Object bean = namedRegistry.get(named.value());
        if (bean == null) throw new IllegalStateException("No bean named '" + named.value() + "'");
        return bean;
    }
    // 原有类型匹配逻辑...
}
```

`@Named` 注解定义：

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.PARAMETER)
public @interface Named {
    String value();
}
```

这是唯一允许的 DI 注解——且只在有歧义时使用。90% 场景类型唯一不需要。
